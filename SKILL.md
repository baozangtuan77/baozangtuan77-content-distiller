# 内容蒸馏（content-distiller）

把外部课程 / 播客 / 博主系列 / 视频 / 长文蒸馏成**一份独立的结构化学习笔记**。

与普通"摘要工具"的区别，在两层别处没有的产出：
1. **交叉对比**——外部观点 vs 你已有的知识库（共识 / 冲突 / 盲区 / 深化）
2. **业务映射**——外部方法论落到你具体的项目、产品、内容选题

这两层由你的 `config.yaml` 驱动。没配置就退化成通用蒸馏；配置了，它就懂你的业务。

---

## 开始前：读取配置

第一步永远是 **Read `config.yaml`**（没有就提示用户从 `config.example.yaml` 复制）。后文所有写 `config.xxx` 的地方，都指这个文件里的对应字段。

---

## 触发方式

`/content-distiller [输入源]`

### 输入源（从常见到少见）

| 优先级 | 输入类型 | 典型来源 | 处理方式 |
|-------|---------|---------|---------|
| **P0** | 原始录音 / 音频文件 | 听课、听行业分享的录音 | 先转写成文字 → 单源蒸馏 |
| **P1** | 一批同主题链接（同一博主系列） | "这个博主近期的内容" | 批量转写 → **合并成一份笔记** |
| **P2** | 单条视频链接（YouTube / B站 / 长视频） | 偶尔一条值得专门蒸馏 | 转写 → 单源蒸馏 |
| **P2** | 长文章链接 / 本地转写稿 / 粘贴文本 | 爆款长文、已有逐字稿 | 直接读 / 抓取正文 → 单源蒸馏 |

> 转写这一步用你自己的工具链（任意 ASR / 字幕提取）。本 skill 从"已有文字"开始。

---

## 执行步骤

### 第 0 步：识别输入 + 补齐信息

转写 / 抓取得到文字后，补齐这些（转写稿里通常抽不到，**找不到就问用户，绝不编**）：
- 讲师 / 博主姓名、身份、实战背景（找不到标 `⚠️ 待验证`）
- 内容正式名称、来源（付费课名 / 账号 / 频道）
- **你听这个的动机**——为了哪个项目 / 哪个方向？**这个问题必须问**，它决定后续业务映射的重心

---

### 第 1 步：多源合并（只有 P1 场景需要）

多条转写稿先合并再蒸馏：
1. 按**主题**聚合，不按时间顺序——同一主题散在不同条里的合到一起
2. 观点去重——反复说的同一方法论，选例证最具体的那版，其余并入"其他例证"
3. 保留出处标注——每个论点注明来自哪条（"源：笔记 3"）
4. 废话 / 自我宣传 / 开场白 / 结尾求赞**全删**

产出一份合并主文本（供后续用，不单独存档）。

---

### 第 2 步：5 维可蒸馏度评分（决定处理深度）

每维 0-5 分：

| 维度 | 判断标准 |
|------|---------|
| **可执行性** | 有具体步骤/清单/模板 = 高；只有口号 = 低 |
| **具体度** | 有数字/案例/真实情境 = 高；全抽象 = 低 |
| **新颖度** | Glob `config.output_dir` 看是否已有同主题。老观点换包装 = 低 |
| **业务相关** | 能映射到你的真实业务（见 `config.business_dimensions`）= 高；纯科普 = 低 |
| **密度** | 单位时长信息量大 = 高；闲聊/广告多 = 低 |

| 总分 | 档位 | 处理 |
|------|-----|-----|
| ≥ 18 | 深度档 | 完整模板（框架+方法论+交叉对比+业务映射+行动清单+金句+风险） |
| 10-17 | 标准档 | 框架+方法论+交叉对比+业务映射（省略行动清单和金句） |
| < 10 | 精简档 | 一屏备忘（标题+3-5 要点），不进正式笔记库 |

**汇报评分 + 建议档位，等用户确认**。用户可覆盖。

---

### 第 3 步：核心框架提取（所有档位）

1. **一句话命题**：作者核心想证明什么？
2. **核心框架**：主干逻辑 3-7 点
3. **讲师画像**：身份 / 资历 / 实战背景 / 客群（用于可信度和适用性判断）

**关键原则**：
- **提取不诠释**——原文没说的不脑补
- **置信度标注**——作者断言没论据的，标 `⚠️ 作者断言，未提供论据`

---

### 第 4 步：关键方法论（深度/标准档）

按"方法论单元"而非"章节顺序"组织。每个方法论独立一节，固定格式：

> **方法名 → 一句话解释 → 核心步骤/要素 → 适用场景 → 原文最具体的例证**

互相关联的，注明"与 XX 配合使用"。

---

### 第 5 步：⚡ 知识库交叉对比（配置驱动 · 核心层之一）

> 需要 `config.knowledge_base_dirs`。没配置就跳过这步并告知用户。

**不只是学外部内容，而是用外部视角审视你自己的认知系统，发现盲区、冲突、验证与深化。**

#### 5.1 选对比库
按外部内容的视角，匹配 `config.knowledge_base_dirs` 里 `compare_when` 命中的目录。混合视角就两类都比。

**先 Glob 命中的目录拿全部 `.md` 文件清单**，再按内容提到的方法论**针对性 Read 5-15 个最相关的**（不要全库读）。

#### 5.2 四象限分类（每条方法论归一类）

```
┌─────────────────┬─────────────────┐
│   ✅ 共识        │   ⚠️ 相反        │
│  外部与你观点一致 │  外部与你观点冲突 │
├─────────────────┼─────────────────┤
│   💡 盲区        │   📈 深化        │
│  外部有而你没有   │  你有但外部讲得更透│
└─────────────────┴─────────────────┘
```

#### 5.3 每类输出格式

**✅ 共识**（验证你认知的正确性）：
```markdown
- **[观点]**
  - 外部说：[原文精炼]
  - 你的知识库：[文件名] 里说 [原文精炼]
  - 一致度：[完全一致 / 大方向一致但细节不同]
```

**⚠️ 相反**（最值得警惕）：
```markdown
- **[冲突点]**
  - 外部说：[A]
  - 你的知识库（[文件名]）说：[B]
  - 冲突本质：[前提不同？客群不同？阶段不同？还是真认知错位？]
  - 建议：[站你一方 / 站外部一方 / 分场景用 / 需进一步验证]
```

**💡 盲区**（最有价值，扩展认知版图）：
```markdown
- **[新概念/新方法]**
  - 外部说：[完整解释]
  - 你的知识库检索结果：**未找到相关内容**（已查 X 个文件）
  - 评估：[真盲区 / 你有意识地不做 / 当前阶段用不上]
  - 若是真盲区：建议补充到 [具体目录下的新文件名建议]
```

**📈 深化**（你已有但外部讲得更透）：
```markdown
- **[概念]**
  - 你的现有认知（[文件名]）：[精炼版]
  - 外部补充的深度：[案例 / 更精细框架 / 边界条件]
  - 建议：是否追加到你的原文件（标注"[来源]补充"）
```

#### 5.4 对比原则
1. **先 Glob 再 Read**：按主题精准定位 5-15 个文件，不全库扫
2. **以你的文件为权威基准**：你的知识库是实战沉淀，不轻易说"你错了"，除非有充分理由
3. **相反观点深挖"为什么相反"**：很多时候不是对错，是前提不同
4. **盲区判断保守**：搜 3-5 个关键词都没找到才算真盲区
5. **不为凑四象限硬分**：点到为止的方法论没必要强行归类

---

### 第 6 步：⚡ 业务映射（配置驱动 · 核心层之二）

> 需要 `config.business_dimensions` + `config.strategy_docs`。**这是这个工具的核心价值，通用蒸馏做不到。**

#### 6.0 映射前必读源文件
按 `config.strategy_docs` 列出的文件，**先读再映射，不要凭印象**。按外部内容主题挑相关的读，不要一次全读。重点抓每个文件 `read_for` 标注的内容。

#### 6.1 逐维度映射
对 `config.business_dimensions` 里每个维度：
- 从 `source_fields` 指向的源文件取真实信息
- 回答该维度的 `mapping_question`
- 命中 `redlines` 的要标 ⚠️

映射必须**具体到项目名、产品功能、文件、版本号**。抽象即无效。

#### 6.2 战略验证（最有价值的一层）
`config.strategy_docs` 指向的文件里通常有"开放问题 / 本周待办"章节：
1. Read 这两节，当 checklist
2. 逐条判断外部内容有无相关论据
3. 能回答 → 详细写怎么回答、证据是什么；不能 → **诚实说不相关**，不硬凑

#### 6.3 阶段约束（硬 filter）
对照 `config.current_stage`：外部内容推的方向若与 `hard_constraints` 冲突 → 标 ⚠️。
典型要标风险的：推"all in 新方向 / 立刻大规模扩张 / 放弃现有业务转型"，而你当前阶段是打磨/收口。

#### 6.4 总判断
用 `config.core_judgment` 里的问题逐条自问，给出"这套内容对我的净贡献"。

---

### 第 7 步：可立即行动清单（深度档必选）

P0-P3 清单。每条**必须可执行**，具体到改哪个文件 / 做哪个动作 / 推哪个项目：

```markdown
| 优先级 | 行动 | 驱动的业务 |
|-------|------|-----------|
| P0 | [最该立刻做的一件] | [具体项目/具体问题] |
```

**反例**（绝不写）：「加强 XX 建设」「优化 XX 体验」「探索新模式」
**正例**：「把 XX 方法论的 3 步框架喂进 [具体文件/prompt]，验证 XX」

---

### 第 8 步：话术金句（深度档选做）

5-10 条可直接二创的金句：原文 + 精炼版同时给，只挑能用在你内容里的。鸡汤/翻译腔/废话不收。

---

### 第 9 步：风险提示 / 不要照搬（深度档必选）

主动标出：
- 作者预设前提与你业务的偏差（地域/行业/阶段/客群）
- 哪些做法在你的场景下会踩坑
- 哪些金句不要复用

---

### 第 10 步：质量校验（所有档位）

写入前自检：
- [ ] **禁用词**（`config.banned_words`）？有就改
- [ ] **空话检查**：每个观点后有具体案例/数字/机制？没有的删
- [ ] **业务映射**具体到 `config.business_dimensions` 里的真实抓手？否则打回
- [ ] **风险提示**没有无脑吹（"非常好/非常棒"）
- [ ] **金句**有论据支撑
- [ ] **讲师信息**纯编的标 `⚠️ 待验证`

---

### 第 11 步：写入文件

**路径**：`config.output_dir` + `config.filename_template`（默认 `{date}-{author}-{topic}.md`）
- 日期 = 蒸馏当天
- 作者 = 讲师真名或 IP 名
- 主题 = 5-12 字精炼，**不照抄原标题**

**顶部元信息（固定）**：
```markdown
# [讲师/博主]·[主题]

> **讲师/博主**：[姓名]（身份/实战背景一行）
> **内容类型**：单课程 / 单视频 / 博主系列（N 条合并）
> **来源**：[URL / 付费课名 / 笔记条目]
> **我的动机**：[为什么听这个]
> **核心命题**：[一句话]
> **与我业务的关联**：[极高/较高/一般/低] —— [一句话为什么]
> **蒸馏评分**：[X/25]（[深度/标准/精简]档）

---
```

---

### 第 12 步：汇报结果

```markdown
## [讲师·主题] 蒸馏完成

**文件路径**：[路径]
**蒸馏评分**：X/25（[档位]）
**与业务关联**：极高 / 较高 / 一般 / 低
**核心命题**：[一句话]
**业务映射要点**：逐维度一句话（不相关的明说"不相关"）
**P0 行动**（如有）：[最值得立刻做的一件]
**风险提示**：[一句话]
```

## 关键原则

1. **提取不诠释**——原文没说的不脑补；断言没论据的标注出来
2. **置信度分级**——强论据（数据/案例）vs 弱论据（观点）要区分
3. **检索优先**——蒸馏前 Glob `config.output_dir` 看同主题是否已有，有就问追加/新建
4. **业务映射必须具体**——到项目、到文件、到版本号，抽象即无效
5. **源文件优先**——映射前先读 `config.strategy_docs`，不凭印象
6. **不被外部叙事倒推**——别因为某大佬说"XX 死了"就觉得你要转型；先看你自己的战略文档

## 踩过的坑

- **不要直接复制转写稿**——笔记是"蒸馏后"的表达，不是"整理后"的逐字稿
- **不要照抄原文标题**——主题自己拟，通常更精炼
- **广告/寒暄/求赞段全删**
- **音译错误手改**——ASR 常把人名/术语听错
- **讲师信息宁可留空也不编**——标 `⚠️ 待验证`
- **博主系列**：合并按主题不按时间，每个论点留出处

# config.yaml 字段说明

复制 `config.example.yaml` 为 `config.yaml` 后，按下表填写。`config.yaml` 已被 `.gitignore`，不会上传——放心写你的真实业务信息。

| 字段 | 类型 | 作用 | 不填会怎样 |
|------|------|------|-----------|
| `output_dir` | 字符串 | 蒸馏产出写到哪个目录 | 必填 |
| `filename_template` | 字符串 | 产出文件命名，可用 `{date}` `{author}` `{topic}` | 用默认 `{date}-{author}-{topic}.md` |
| `banned_words` | 列表 | 你不想出现的"正确废话"词，蒸馏时自动检查改写 | 跳过禁用词检查 |
| `strategy_docs` | 列表 | 战略/全景文档路径，蒸馏时读取用于业务映射与战略验证 | 跳过"战略验证"层，映射会变弱 |
| `current_stage` | 对象 | 当前阶段名 + 硬约束，用于过滤与当前阶段冲突的外部建议 | 跳过阶段过滤 |
| `knowledge_base_dirs` | 列表 | 你的知识库目录，蒸馏时按主题对比找共识/冲突/盲区/深化 | 跳过"交叉对比"层 |
| `business_dimensions` | 列表 | **核心**。你的真实业务维度，外部内容逐条映射回这些维度 | 退化成通用蒸馏，失去差异化 |
| `core_judgment` | 列表 | 映射时反复自问的战略锚点问题 | 跳过总判断 |

## 两个"层"是可选的，但它们才是这个工具的价值所在

- **不填 `knowledge_base_dirs` / `business_dimensions`** → 工具退化成一个普通的"课程/内容蒸馏器"：能提取框架、方法论、金句，但**不会**告诉你"这对我手上的项目有什么用"。
- **填了** → 每次蒸馏都会多出两层别处没有的产出：
  1. **交叉对比**：外部观点 vs 你已有的知识库 → 共识 / 冲突 / 盲区 / 深化四象限
  2. **业务映射**：外部方法论 → 落到你具体的客户、项目、产品功能、内容选题

把你私有的业务"声明"在 config 里，引擎就成了你的私人内容参谋，而不是一个通用摘要工具。

## 子字段细节

### `strategy_docs[].read_for`
告诉引擎"这个文件读出来干嘛用"。引擎会优先找文件里的 **"开放问题" / "本周待办" / "演进路径"** 之类章节，用外部内容去回答它们。

### `business_dimensions[]`
- `name`：维度名（如"B端服务""C端产品""个人IP"）
- `source_fields`：这个维度的真实信息从哪来（引擎映射前会去 `strategy_docs` 里找对应内容，不靠臆测）
- `mapping_question`：映射时必须回答的问题（越具体，产出越落地）
- `redlines`：这个维度的红线，命中要标 ⚠️（如"已做完的别建议重做"）

### `knowledge_base_dirs[].compare_when`
判断条件。引擎识别外部内容是什么视角后，只对比 `compare_when` 命中的目录，避免全库乱扫。

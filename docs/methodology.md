# 为什么"业务映射"比通用蒸馏强 10 倍

> 这是 content-distiller 背后的方法论。也是它和市面上所有"AI 摘要/读书笔记"工具最大的不同。

## 一个被忽略的事实：摘要是负资产

你听完一节课，让 AI 给你一份摘要。它写得很工整：核心观点 5 条、金句 8 句、思维导图一张。

然后呢？

然后它躺在你的笔记软件里，再也没被打开过。

因为**摘要回答的是"这节课讲了什么"，而你真正的问题是"这对我有什么用"**。这两个问题之间隔着一条鸿沟，通用工具跨不过去——因为它不知道"你"是谁、在做什么、卡在哪。

## 蒸馏的真正价值，在两次"对撞"

一份有用的笔记，不是把外部内容整理干净，而是让它和**你已有的东西**发生两次对撞：

### 对撞一：和你的知识库对撞

你脑子里、文件里已经有一套认知。外部内容进来，应该被逼着回答四个问题：

- **它和我已知的一致吗？**（✅ 共识——验证你没错，这很重要，能让你更敢下注）
- **它和我已知的冲突吗？**（⚠️ 相反——最值得停下来想的，到底是谁错了，还是前提不同）
- **它讲了我完全没有的东西吗？**（💡 盲区——这是新内容里最值钱的部分）
- **它把我已有的讲得更透了吗？**（📈 深化——可以回去补强你的原有认知）

没有这次对撞，你只是"知道了一个新观点"；有了这次对撞，你是"更新了你的认知系统"。

### 对撞二：和你的真实业务对撞

知道了不等于有用。第二次对撞，是把外部方法论按在你**具体的项目、产品、内容**上：

- 这能用在我哪个项目的哪一步？
- 这能改进我产品的哪个功能？
- 这能成为我哪条内容的选题？
- 这能回答我战略上正卡着的哪个开放问题？

判断标准只有一个：**能不能具体到项目名、文件、版本号**。说不出具体的，就是没用。

## 为什么通用工具做不到这一层

因为这两次对撞都需要一样东西：**你的私有上下文**。你的知识库、你的项目、你的战略、你的红线。

通用工具不可能内置这些——它们也不该内置（那是你的隐私）。

所以 content-distiller 的做法是：**把私有上下文外置成一个 config 文件**。

```
通用蒸馏引擎（开源、人人可用）
        +
你的 config.yaml（私有、永不上传）
        =
懂你业务的私人内容参谋
```

引擎是公开的，业务是你的。你声明一次"我的知识库在哪、我的业务有哪几条线、我的红线是什么"，之后每一次蒸馏，AI 都自动带着这套上下文去对撞。

## 几条硬原则

做这件事，有几条原则是反"AI 默认行为"的，必须明确写进规则里：

1. **提取不诠释**——AI 特别爱脑补。原文没说的，一个字都不许加。作者断言但没给论据的，标出来。
2. **以你为权威基准**——你的知识库是你实战沉淀的。不要因为某个大佬说了什么，就轻易判你"错了"。先问是不是前提不同。
3. **不被外部叙事倒推**——这是最隐蔽的坑。看到"XX 模式已死"就慌着转型，是把别人的剧本套在你身上。先回去看你自己的战略文档。
4. **抽象即无效**——"加强建设""优化体验""探索模式"这种话，等于没说。每条行动必须能落到一个具体动作。

## 一句话总结

> 通用工具帮你**省时间**（不用自己读）。
> 这个工具帮你**做决策**（读完知道该改哪个项目、补哪个盲区、写哪条选题）。

省时间的工具到处都是。帮你做决策的，得懂你。而"懂你"这件事，只能由你自己用一个 config 喂给它。
